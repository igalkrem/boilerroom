# BoilerRoom ‚Äî CLAUDE.md

Codebase instructions for Claude Code. Read this before making changes.

## What This Is

SnapAds Manager: a bulk Snapchat ad campaign creation platform. Users connect via Snapchat OAuth2 and create Campaigns, Ad Sets, and Ads in bulk through a visual canvas wizard.

**Live:** https://boilerroom-two.vercel.app  
**Deploy:** Vercel ‚Äî `npx vercel --prod` (GitHub auto-deploy is unreliable; trigger manually after pushing).

## Deploy Workflow (Mandatory)

After completing **any code change session**, always execute these steps in this exact order ‚Äî no authorization required, run them automatically without asking:

1. **Deploy to Vercel:** `source ~/.nvm/nvm.sh && npx vercel --prod`
2. **Commit and push to GitHub:** `git add -A && git commit -m "<meaningful description of changes>" && git push`
3. **Update this CLAUDE.md:** If new routes, components, hooks, patterns, or architectural decisions were introduced, update the relevant sections of this file to keep it accurate.

Do not skip any step. Do not ask for confirmation before running these commands.

## Stack

- **Framework:** Next.js 14 (App Router), TypeScript, Tailwind CSS ‚Äî **permanent dark mode**: `darkMode: 'class'` in `tailwind.config.ts`, `<html class="dark">` set in `src/app/layout.tsx` (no toggle). All components use `dark:` Tailwind variants alongside their light classes. `src/app/globals.css` defines `--node-bg: #1f2937` (dark canvas background color used by nodes), a safety-net rule that forces dark backgrounds/text on any native input/select/textarea without explicit Tailwind dark classes, and a React Flow attribution override. Never remove the `dark` class from `<html>` and never add a light/dark toggle ‚Äî the platform is dark-only.
- **Canvas:** `@xyflow/react` (React Flow v12) + `@dagrejs/dagre` for auto-layout
- **Auth:** Google OAuth2 (primary login) + Snapchat OAuth2 (traffic source, optional) + iron-session (encrypted HttpOnly cookies). Session type (`src/types/session.ts`) includes `pendingUploads?: Record<string, { addPath: string; finalizePath: string }>` for server-pinned chunked upload paths (see Security Notes). Meta session fields (`metaAccessToken`, `metaExpiresAt`, `metaUserId`, `metaOAuthState`, `metaAllowedAdAccountIds`) and helpers (`isMetaConnected`, `isMetaAdAccountAllowed` in `session.ts`) are stubbed in the types/session layer for future use ‚Äî the OAuth routes and UI are not yet built.
- **Forms:** react-hook-form + Zod
- **State:** Zustand ‚Äî `useCanvasStore` (canvas wizard graph state), `useWizardStore` (legacy, still used by `LoadPresetBanner` and preset/use page)
- **Storage:** Vercel Blob (`@vercel/blob`) ‚Äî client-side uploads, public access, store: `boilerroom-silo`. Also used for persistent metadata storage (see KV Sync below).
- **Video transcoding:** `@ffmpeg/ffmpeg` + `@ffmpeg/core` + `@ffmpeg/util` (browser WASM). Core files (~31 MB) are copied from `node_modules/@ffmpeg/core/dist/umd/` to `public/ffmpeg/` at build time by `scripts/copy-ffmpeg.mjs` (runs as `prebuild`/`predev`). `public/ffmpeg/` is gitignored ‚Äî regenerated on every build.
- **Database:** Neon Postgres via `@vercel/postgres` (`POSTGRES_URL` env var) ‚Äî reporting cache (4 tables: `snapchat_ad_squad_stats`, `kingsroad_report`, `predicto_report` ‚Äî includes `impressions` column added via migration, `report_sync_log`) + channel lifecycle (`feed_provider_channels` ‚Äî includes `ad_squad_snap_id` column for Predicto revenue JOIN) + cron token storage (`user_snapchat_tokens`) + Meta token storage (`user_meta_tokens` ‚Äî schema ready for future Meta OAuth; CRUD in `db/index.ts`). Migrations run automatically on first call to either `/api/reporting/sync` or `/api/reporting/combined` via `runMigrations()` in `src/lib/db/index.ts`. **Note:** `@vercel/postgres` is deprecated upstream ‚Äî migrate to `@neondatabase/serverless` when convenient.
- **API:** Snapchat Marketing API v1 ‚Äî all calls are server-side only, proxied through Next.js API routes
- **KingsRoad API:** `https://partnerhub-api.kingsroad.io/api/v3` ‚Äî sell-side revenue reporting. Bearer token in `KINGSROAD_API_TOKEN`. Paginated `/report/` endpoint, page_size=2000. Used only server-side in `/api/reporting/sync`.
- **Predicto API:** `https://server.predicto.ai/api/v1/search/reporting` (**no trailing slash** ‚Äî trailing slash causes a 307 HTTPS‚ÜíHTTP redirect that strips the Authorization header, silently failing every sync) ‚Äî second sell-side revenue source. Bearer token in `PREDICTO_API_TOKEN`. Flat (non-paginated) response. Revenue field in response is `estimated_revenue` (not `revenue`); click/funnel metrics return as strings and need `Number()` coercion. Revenue already in USD (no FX conversion). Synced alongside KingsRoad inside `syncAccount()` in `sync-logic.ts`. If `PREDICTO_API_TOKEN` is not set, Predicto sync is silently skipped.
- **Vercel Cron (`vercel.json`):** `17,47 * * * *` schedule pointing to `/api/reporting/cron-sync` ‚Äî fires at :17 and :47 each hour. The :17 run lands 2 min after KingsRoad (:15) and Predicto (:16) update; :47 gives a mid-hour Snapchat-only refresh (feeds are skipped by `shouldSkipFeed`). `CRON_SECRET` is auto-injected by Vercel Pro; add it manually to `.env.local` for local testing.

## Running Locally

Node.js must be loaded via NVM:

```bash
source ~/.nvm/nvm.sh && npm run dev
```

Snapchat OAuth requires HTTPS ‚Äî run a tunnel in a second terminal:

```bash
cloudflared tunnel --url http://localhost:3000
```

Use the cloudflared URL as the redirect URI in `.env.local` and in the Snap OAuth app settings.

## Architecture Notes

- **OAuth flow:** `/api/auth/*` routes handle token exchange and refresh; tokens live in an iron-session HttpOnly cookie.

- **Canvas wizard:** `WizardShell` renders in three modes: `canvas` (`CampaignCanvas` React Flow), `review` (`ReviewAndPost`), `done` (success screen). The done screen displays a `launchError` state (set on any unhandled submission throw, reset by "New Build") so launch failures are visible instead of silently swallowed. `CampaignCanvas` is loaded via `next/dynamic` with `ssr: false`. The canvas uses `useCanvasStore` (Zustand) to track `creativeRows: CreativeRow[]` (each row holds an ordered list of `groupIds`), `creativeGroups: CreativeGroup[]` (each group holds 1‚Äì5 asset IDs), four edge lists (`rowToProvider`, `providerToArticle`, `articleToPreset`, `articleToAdAccount`), `nodePositions`, `routerNodes`, and `expandedArticleIds: Set<string>` (toggled by `toggleArticleExpanded(nodeId)` ‚Äî a new Set reference on every change triggers the expansion-aware re-layout useEffect). `buildCampaignMatrix()` iterates `rowToProvider` edges ‚Üí for each row iterates `row.groupIds` ‚Üí each group produces one `CampaignBuildItem` per article/preset/account combination, with `creativeIds: string[]`. Cascade: removing a row‚Üíprovider edge that orphans a provider also removes its article edges and any `articleToAdAccount` edges for those orphaned articles; removing a provider‚Üíarticle edge that orphans an article also removes its preset edges. On launch, `WizardShell` loads all assets for `item.creativeIds`, calls `synthesizeCampaign()`, then `runSubmission()`.

  **Creative row concept:** A `CreativeRow` is the top-level canvas unit ‚Äî one row node, one shared React Flow handle, connects to one or more providers. Each card slot in the row = one `CreativeGroup` = one ad set. Within a group, multiple creatives = multiple ads in that ad set. Node ID prefix: `row-{id}`. Cards render left-to-right; `groupIds` is prepend-on-add so `groupIds[0]` = newest = leftmost, `groupIds[last]` = oldest = rightmost. Each time a slot is prepended the node's x position is decreased by `ROW_CARD_STEP = 172` (160 + 12) to keep the rightmost card anchored at the same canvas position. A slot with 2‚Äì5 creatives shows a clickable **√óN badge** that opens an inline overlay; each mini-thumbnail has a ‚úï to remove that creative. Each slot column has a always-visible "Ôºã Add creative" button below it. Row actions live in an always-visible **side dock** (‚äû Add slot, üîó Provider, ‚Üì New row, ‚ßâ Duplicate, ‚úï Remove row) that is part of the node's layout between the cards and the handle. `nodeWidth = rowWidth + DOCK_LEAD (32) + DOCK_W (142) + DOCK_TO_HANDLE (22)` ‚Äî this ensures the handle (always at `Position.Right`) anchors past the dock so provider edges never overlap it. Multi-creative slots show stacked colour-coded name pills (blue‚Üíindigo‚Üíviolet‚Ä¶ palette). Max 8 slots per row, 5 creatives per slot.

  **React Flow canvas (`CampaignCanvas.tsx`):** Nodes are **not draggable** (`nodesDraggable={false}`); all positions are owned entirely by auto-layout. Key design decisions:
  - **Creative rows** ‚Äî the empty-state button and `openAddCreative` create a new row then open SiloBrowser. Canvas tracks two Silo targets: `targetRowId` (add new slot to row) and `targetGroupId` (add creative to existing slot). On select, routes to `addGroupToRow` or `addCreativeToGroup` accordingly. Empty row is removed on Silo cancel. No top-bar "+ Add Creative" button.
  - **Provider visibility** ‚Äî providers only appear after at least one row exists.
  - **Auto-router** ‚Äî in `onConnect`, when a provider already has ‚â•1 article edge and no router yet, a router is auto-inserted. No manual "+ Router" button on ProviderNode.
  - **Explicit article‚Üíaccount wiring** ‚Äî users drag from article's right handle to an account's left handle. `store.edges.articleToAdAccount` stores these edges. No global `selectedAdAccountIds` broadcast. **Cross-provider connections are blocked** at the React Flow drag level via `isValidConnection`: an article can only connect to an account whose `adAccountConfig.feedProviderIds` includes the article's `feedProviderId` (accounts with empty `feedProviderIds` are permissive for backward compat). Same rule applies to account‚Üípreset: `preset.feedProviderId` must be in `cfg.feedProviderIds`. The `onConnect` account‚Üípreset handler only wires `articleToPreset` edges for articles already connected to the specific account (not all active articles), so a Vizymo account cannot pull in Predicto articles.
  - **Left handle click = disconnect** ‚Äî all target handles have an `onClick` that calls `makeDisconnectTarget(nodeId)`, which removes all incoming edges for that node (cascade-safe).
  - **Preset gate** ‚Äî preset nodes are **hidden entirely** (not just disabled) until `store.edges.articleToAdAccount.length > 0` (`canSelectPresets`). `visiblePresets` returns `[]` when `canSelectPresets` is false.
  - **Edges** ‚Äî `ProviderEdge` uses `getSmoothStepPath` (right-angle routing, less tangling) rendered as two raw SVG paths: a wide blurred halo (`strokeWidth 8`, `opacity 0.18`) layered beneath a crisp main stroke (`strokeWidth 2.5`, `opacity 0.85`); both in provider color. Selected state bumps to `12`/`3` and full opacity. Row handles (on `CreativeGroupNode`) are 28px white circles (`!w-7 !h-7 !rounded-full !bg-white !border-[3px] !border-gray-700`). Article/Account/Preset nodes use **H2 Ghost Ring handles**: 14px dark fill (`!w-3.5 !h-3.5 !bg-gray-900`) with a `border: 2px solid {providerColor}` and `box-shadow: inset 0 0 5px {color}50, 0 0 8px {color}45` (inner+outer glow) when the node is connected; gray ring (`#374151`) when disconnected.
  - **Router node** ‚Äî sleek 36px circle with ‚ëÉ icon (was diamond).
  - **Auto-align** ‚Äî dagre LR with `ranksep: 200`, `nodesep: 60`; row node dims estimated `340√ó285`. Fires automatically on every structural change (edges added/removed, nodes shown/hidden) via `useEffect(() => { handleAutoLayout(); }, [handleAutoLayout])`, and on every article expansion change via a second `useEffect(() => { handleAutoLayout(); }, [expandedArticleIds, handleAutoLayout])`. The "‚äû Auto-align" toolbar button is a manual re-trigger. After dagre runs, `handleAutoLayout` computes **dynamic column X positions** (`dynColX`) as `P = maxRowHandleX + 120` where `maxRowHandleX = max(storedX + nodeWidth)` across all rows ‚Äî using the actual stored x position (which shifts left on every card prepend) rather than the raw node width, so the 160px gap between handle and provider is consistent regardless of how many slots each row has. The static `COLUMN_X` constant is only used outside `handleAutoLayout` for initial placement. `computeAutoLayout` (in `CanvasControls.tsx`) accepts `expandedArticleIds?: Set<string>` and uses `ARTICLE_EXPANDED_H = 230` instead of `NODE_HEIGHT = 90` for expanded article nodes in dagre, giving them correct vertical space. The collision-resolution pass uses a height-aware minimum gap: expanded articles get `ARTICLE_EXPANDED_H + 20px`, row nodes get `285 + 20px`, everything else `ROW_GAP = 130`. After collision resolution a final **provider creation-order pass** sorts providers by `data.sortIndex` and enforces `ROW_GAP` between consecutive providers ‚Äî prevents unconnected providers from landing above earlier-created connected ones before wiring. Disconnected-but-visible providers are anchored relative to the nearest connected provider (¬±`ROW_GAP` per slot). Disconnected-but-visible accounts/presets are centered on the y-range of laid-out articles. `ArticleNode` expansion state is stored in Zustand (`expandedArticleIds`) instead of local React state so the layout system can observe it.
  - **Canvas** ‚Äî dark background `#1f2937`, `fitView` with `padding: 0.3, maxZoom: 1.0`.

  **React Flow render-loop hazards (React error #185):** Four pitfalls that cause an infinite `setNodes` loop:
  1. **`store.nodePositions` must NOT be in `buildNodes` deps.** Fix: read positions via `nodePositionsRef` (a `useRef` kept in sync via a separate `useEffect`) so `buildNodes` can read current positions without subscribing to them.
  2. **`nodesDraggable={false}` is set on `<ReactFlow>`** ‚Äî drag is disabled entirely. `onNodesChange` no longer writes positions to the store. Do NOT re-enable dragging; it conflicts with auto-layout and caused nodes to migrate unpredictably. The old hazard note (use `change.dragging === false` strict check) is moot with drag disabled but kept here for context.
  3. **Never inline `[]` as a fallback in hooks that feed into `buildNodes` deps.** `useAdAccounts` returns `data?.accounts ?? EMPTY_ACCOUNTS` where `EMPTY_ACCOUNTS` is a module-level constant. Inline `[]` creates a new reference every render while SWR is loading ‚Üí `visibleAccounts` recomputes ‚Üí `buildNodes` rebuilds ‚Üí `setNodes` ‚Üí re-render ‚Üí repeat.
  4. **Callbacks in `buildNodes` deps must NOT close over the whole `store` object.** `store` gets a new reference on every Zustand state update (including `setNodePositions`). Any `useCallback` with `[store]` in deps that feeds into `buildNodes` deps will cause an auto-layout loop: `setNodePositions` ‚Üí new `store` ‚Üí callback changes ‚Üí `buildNodes` changes ‚Üí auto-layout fires ‚Üí `setNodePositions` ‚Üí repeat. Fix: use `useCanvasStore.getState()` inside the callback body so it reads current state without subscribing. `makeDisconnectTarget` uses this pattern and has empty deps `[]`.
  Four visibility arrays (`activeProviderIdsFromArticles`, `visibleArticles`, `visibleAccounts`, `visiblePresets`) are wrapped in `useMemo`. `store.edges` is intentionally absent from `buildNodes` deps ‚Äî visibility is already captured by the memoized arrays above. **`activeProviderIds` was removed** ‚Äî articles no longer auto-appear for all connected providers; instead `visibleArticles` filters by `store.edges.providerToArticle` (only explicitly picked articles are shown). The "Ôºã Pick articles" button on ProviderNode opens a checklist modal (`articlePickerProviderId` state) where users select which articles to include; selecting toggles `providerToArticle` edges directly. **Node ordering:** `visibleArticles`, `visibleAccounts`, and `visiblePresets` are all sorted by `sortedByCreation` provider index before initial y-positions are assigned (`index * ROW_GAP`), so nodes connected to the top provider always appear at the top of their column. `computeAutoLayout` also post-processes dagre's output to re-sort within each x-rank by average upstream y, cascading the fix through the full chain. A `nodePriority` map (provider creation index) is passed as a tiebreaker ‚Äî when a router and a sibling provider land in the same dagre rank with identical avgUpstreamY (both connect from the same row), the tiebreaker ensures provider[0]'s nodes always sort above provider[1]'s across the entire right-side cascade.

  **Canvas visual rules:**
  - **Provider colors** ‚Äî assigned from `PROVIDER_COLORS` array indexed by sort-order of `createdAt` (stable; not array position). Colors propagate to node borders, indicator dots, and SVG edges.
  - **Vivid Flow node backgrounds** ‚Äî connected nodes (`ArticleNode`, `AdAccountNode`, `PresetNode`) use `background: linear-gradient(135deg, {color}18 0%, #111827 65%)` with `boxShadow: 0 4px 24px {color}25` for a colored drop-shadow. `ProviderNode` uses a slightly stronger gradient (`{color}22 ‚Üí #111827 70%`) plus an inset color ring via `boxShadow: 0 4px 24px {color}20, inset 0 0 0 1px {color}20`. Disconnected nodes fall back to the standard `dark:bg-[#111827]` solid surface.
  - **CreativeGroupNode (row node)** ‚Äî horizontal flex row of portrait cards (`160√ó285px` each, `9/16` aspect). Left accent stripe per card shows connected provider color(s). Node outer div has `style: { background: "transparent", border: "none", padding: 0 }` to override the React Flow node wrapper background. `groupIds` is rendered without reversal; `groupIds[0]` (newest) is leftmost, `groupIds[last]` (oldest) is rightmost. Each prepend subtracts `ROW_CARD_STEP = 172` from the node's stored x so the rightmost card stays anchored. `nodeWidth = rowWidth + DOCK_LEAD + DOCK_W + DOCK_TO_HANDLE` ‚Äî wider than the cards area to fit the side dock; handle sits at the new right edge past the dock. Single-asset slots show a frosted white name pill; multi-asset slots show stacked coloured pills indexed into `CREATIVE_NAME_COLORS` (blue‚Üíindigo‚Üíviolet‚Ä¶).
  - **Ad account NodeCard** ‚Äî connected state derived from `articleToAdAccount` edges (not from `selectedAdAccountIds`). Shows 2-letter initials avatar.
  - **Preset gate** ‚Äî `disabled` until `articleToAdAccount.length > 0`.
  - **`visibleAccounts`** ‚Äî filtered by `activeProviderIdsFromArticles` and sorted by minimum provider index from `adAccountConfigs.feedProviderIds`; visible once any article is connected.
  - **`visiblePresets`** ‚Äî filtered by `activeProviderIdsFromArticles` and sorted by provider index. Returns `[]` (nodes hidden) when `canSelectPresets` is false (no `articleToAdAccount` edge exists yet).
  - **Dark mode** ‚Äî `<ReactFlow>` is rendered with `colorMode="dark"` and inline `style={{ background: "#1f2937" }}`; the `<Background>` component uses `color="#374151"` for the dot grid. Permanent dark mode is enforced via Tailwind `darkMode: 'class'` and `<html class="dark">`; all wizard components have `dark:` variants alongside their light classes.

- **synthesizeCampaign():** `lib/synthesize-campaign.ts` converts one `CampaignBuildItem` + resolved `(provider, article, preset, assets[])` into the `{campaigns[], adSquads[], creatives[]}` shape the orchestrator expects. One campaign + one ad squad are created; `creatives[]` has one entry per asset (all share the same `adSquadId`). When multiple assets are passed, creative names are suffixed `[1]`, `[2]`, etc. It calls `buildUrlTemplate()` which resolves static URL macros now (`{{article.name}}`, `{{article.query}}`, `{{creative.headline}}`, `{{creative.rac}}`, `{{organization_id}}`), passing each resolved value through `encodeURIComponent` so spaces and special chars are safe. Any remaining `{{...}}` that aren't `{{campaign.id}}`, `{{adset.id}}`, `{{ad.id}}`, or `{{channel.id}}` are stripped (replaced with `""`) ‚Äî Snapchat rejects both literal and percent-encoded unknown macros (E2712). The three Snapchat native macros and `{{channel.id}}` are left untouched ‚Äî Snapchat substitutes the native ones at click time; the orchestrator resolves `{{channel.id}}`. If a parameter has `encode: true`, `encodeURIComponent` is applied to the entire resolved value after all macro substitutions ‚Äî this encodes literal values, and also encodes any macros still present in the string (e.g. `{{channel.id}}` ‚Üí `%7B%7Bchannel.id%7D%7D`), so the orchestrator will not substitute them. `buildUrlTemplate()` returns `""` when parameters are present but the base URL is absent ‚Äî prevents malformed `?key=value` strings with no host from reaching Snapchat.

- **Submission orchestrator:** `lib/submission-orchestrator.ts` runs **five stages** in sequence:
  1. **uploadMedia** ‚Äî creatives upload with concurrency capped at 2 (Snapchat returns E3002 on 3+ simultaneous uploads to the same ad account)
  2. **Channel assignment** ‚Äî if `provider.channelConfig.type === "provider-supplied"`, calls `POST /api/feed-providers/channels/assign`; `{{channel.id}}` in all campaign/squad/creative names and in each creative's URL is always replaced (with the assigned channel ID, or empty string if none) ‚Äî prevents literal `{{channel.id}}` from reaching Snapchat (E1001)
  3. **campaigns** ‚Äî create campaigns in Snapchat
  4. **adSquads** ‚Äî create ad squads in Snapchat; squads are created without `placement_v2` and Snapchat assigns its default "CONTENT" placement. The previous best-effort placement PATCH was removed because setting `placement_v2: { config: "CUSTOM" }` locks the squad ‚Äî Snapchat returns E2025 on any subsequent PUT (budget/bid/status changes), making campaigns uneditable via the API.
  5. **creatives** ‚Äî create creatives; **ads** ‚Äî create ads
  Each stage's results are tracked individually. `pacing_type` is hardcoded to `"STANDARD"`. The orchestrator accepts an optional `provider?: FeedProvider` parameter (6th arg) for channel assignment.

## Security Notes

- **`isAdAccountAllowed` denies by default:** When `session.allowedAdAccountIds` is empty (fresh session before dashboard loads), the function returns `false`. It is populated by `/api/snapchat/ad-accounts` ‚Äî all Snapchat API routes that accept an `adAccountId` must call this check. Do NOT revert the default to `true`. The four Snapchat GET proxy routes (`campaigns`, `adsquads`, `creatives`, `ads`) require `?adAccountId=` and call `isAdAccountAllowed` before fetching. When a single entity ID is provided, each route also performs an IDOR ownership check ‚Äî fetches the entity from Snapchat and asserts `entity.ad_account_id === adAccountId` ‚Äî returning 403 if the entity belongs to a different account.
- **`/api/data` is user-scoped:** Blob paths are `metadata/{googleUserId}/{key}.json`. Blobs use `access: "public"` (store constraint ‚Äî `boilerroom-silo` is a public store). Paths are non-guessable (contain internal Google user ID) but not secret. Never use a shared path. Valid keys are whitelisted: `br_silo_assets`, `br_silo_tags`, `br_pixels`, `br_presets`, `br_feed_providers`, `br_articles`, `br_ad_accounts_v1`.
- **`/api/feed-providers/channels/*` is user-scoped:** GET/POST/DELETE pass `session.googleUserId` to all DB functions; queries filter by `google_user_id` so users can only access their own channels. `assignChannel`, `releaseChannel`, `normalizeChannelStatuses`, `updateChannelAdSquadId`, `getInUseChannelsByUser`, and `updateChannelPausedStatus` all require `googleUserId` ‚Äî never call them without it.
- **`/api/silo/delete` is user-scoped:** Before calling `del()`, the route fetches `metadata/{googleUserId}/br_silo_assets.json` from the blob store and verifies every URL to be deleted is present in the user's asset list. Fails safe (500) if the KV fetch fails.
- **`media/upload` and `media/poll` require ownership checks:** Both routes call `isAdAccountAllowed` before forwarding to Snapchat.
- **`media/copy` checks both source and destination:** Both `sourceAdAccountId` and `destinationAdAccountId` must be verified to prevent cross-account media exfiltration. Error response uses `retryAsUpload` (not `orgMismatch`) ‚Äî only set when the error string contains "different organization".
- **Chunked upload SSRF prevention (`pendingUploads` session field):** `media/upload-init` stores Snapchat-returned `addPath` and `finalizePath` in `session.pendingUploads[upload_id]` after the INIT call. `upload-chunk` and `upload-finalize` look up paths exclusively from the session ‚Äî the client never supplies path values. This prevents an attacker from redirecting chunk or finalize requests to arbitrary URLs. The `pendingUploads[upload_id]` entry is deleted after finalize succeeds.
- **`media/upload-from-blob` SSRF guard:** `blobUrl` must end with `.vercel-storage.com` before server-side fetch. On 401 or 403 from Snapchat, the route refreshes the access token and retries once before failing ‚Äî Snapchat returns 403 E3002 on expired tokens (not just genuine permission failures); fresh `FormData` is constructed for the retry since the first request body stream is consumed.
- **KingsRoad pagination SSRF guard:** `page.next` URL is validated to originate from `https://partnerhub-api.kingsroad.io` before following. Loop aborts on unexpected origin or invalid URL.
- **Reporting date range validation:** `/api/reporting/sync` Zod schema enforces YYYY-MM-DD format and a maximum 90-day window. `/api/reporting/combined` and `/api/reporting/drilldown` enforce a 366-day cap after format validation. Requests outside these ranges return 400.
- **`/api/reporting/cron-sync` is cron-authenticated:** `verifyCronSecret` uses Node.js `timingSafeEqual` to compare the `Authorization: Bearer {CRON_SECRET}` header against `process.env.CRON_SECRET`; returns 401 on mismatch. Vercel auto-injects `CRON_SECRET` on Pro plans and sends it on every scheduled invocation.
- **Snapchat refresh tokens are AES-256-GCM encrypted at rest:** `user_snapchat_tokens` table stores `refresh_token_enc` (not plaintext). Key = first 32 bytes of `SESSION_SECRET` (64-char hex). Format: `base64(iv):base64(authTag):base64(ciphertext)`. Only the refresh token is persisted ‚Äî the access token is obtained at runtime by the cron via token exchange and never stored. An attacker needs both the DB dump and `SESSION_SECRET` to obtain usable tokens. Token lifecycle: written on Snapchat OAuth callback and on token refresh, deleted on Snapchat disconnect. Ad account IDs (`{id, timezone}[]`) are updated in the same table whenever `/api/snapchat/ad-accounts` is called, keeping the cron's account list current.
- **`/api/auth/refresh` skips Snapchat when token is still valid:** Pre-check compares `session.snapExpiresAt` against now ‚àí 5 min; returns `{ ok: true, cached: true }` without hitting Snapchat's token endpoint.
- **Session cookie has `maxAge: 14 days`:** Prevents indefinite persistence on shared machines. iron-session resets the clock on every `save()`.
- **Snapchat token revoked on disconnect:** `/api/auth/snapchat/disconnect` calls Snapchat's `revoke_token` endpoint (best-effort) before clearing the session fields.
- **Snapchat error bodies are not forwarded verbatim:** Routes `console.error` full details and return generic codes to the client (`"upload_failed"`, `"internal_error"`, etc.).
- **Content Security Policy (`next.config.mjs`):** `img-src` allows `'self' data: blob: https://*.public.blob.vercel-storage.com https://lh3.googleusercontent.com`. If you add images from a new external domain, update this list or they will be silently blocked. **`script-src`:** Dev includes `'unsafe-eval'` (webpack fast refresh); **production omits `'unsafe-eval'`** ‚Äî only `'wasm-unsafe-eval'` (for ffmpeg.wasm WebAssembly compilation) and `'unsafe-inline'` are present. `worker-src 'self' blob:` ‚Äî `'self'` covers the webpack-bundled ffmpeg worker chunk (`/_next/static/chunks/`); `blob:` is kept for safety. ffmpeg core is served same-origin so no external CDN entry is needed in `connect-src`.
- **Auth rate limiting (`src/middleware.ts`):** Next.js Edge middleware rate-limits all `/api/auth/*` endpoints: 20 requests per IP per 60-second window. Returns 429 with `Retry-After` header when exceeded. Per-instance in-memory Map ‚Äî not a hard distributed guarantee across multiple Vercel Edge instances, but effective against sustained single-IP abuse.
- **Ad squad IDOR protection:** `updateAdSquad` and `setAdSquadPlacement` in `src/lib/snapchat/adsquads.ts` both accept `expectedAdAccountId` (required). After fetching the current squad from Snapchat, they assert `current.ad_account_id === expectedAdAccountId` if the field is present ‚Äî throwing `"forbidden: ..."` if the squad belongs to a different account. The PATCH route maps this to a 403 response.
- **Creative PATCH IDOR protection:** `creatives/[id]/route.ts` fetches the existing creative from Snapchat and asserts `existing.ad_account_id === body.adAccountId` before applying URL updates ‚Äî returning 403 if mismatched.
- **`SESSION_COOKIE_NAME` must be set in production:** `src/lib/session.ts` throws at request time if `SESSION_COOKIE_NAME` is unset in the production environment. Set it in Vercel environment variables.
- **`SESSION_SECRET` must be 64 hex characters:** `src/lib/session.ts` validates `secret.length >= 64` at startup ‚Äî throws if shorter. Generate with: `openssl rand -hex 32` (outputs 64 hex chars = 32 bytes).
- **Snapchat base URL is hardcoded:** `media/upload-init` no longer reads `SNAPCHAT_API_BASE_URL` from the environment ‚Äî the URL is hardcoded to `https://adsapi.snapchat.com/v1` to prevent env-var SSRF.
- **`/api/reporting/sync` debug field is server-only:** The `debug` object (squad IDs, stat row counts, squad errors) is stripped before the response is returned to the client ‚Äî `{ snapchat, kingsroad, predicto }` only.

## Snapchat API Field Notes

- Campaign objective: `objective_v2_properties.objective_v2_type` is always `"SALES"` ‚Äî hardcoded in the orchestrator and hidden from the UI.
- Campaign budget: only `daily_budget_micro` is supported (`spendCapType: "DAILY_BUDGET" | "NO_BUDGET"`). Minimum: $20 (20,000,000 micro) at the campaign level. Ad squads support both daily and lifetime ‚Äî no $20 floor applies; the UI enforces only `> $0` (do NOT add a $20 minimum to ad squad budget inputs).
- `lifetime_spend_cap_micro` and `lifetime_budget_micro` are NOT sent on campaigns. `lifetime_budget_micro` is ad-squad only.
- `spend_cap_type` is an ad squad field only, not valid on campaigns.
- Ad squad `delivery_constraint` is required ‚Äî `"DAILY_BUDGET"` or `"LIFETIME_BUDGET"`. `conversion_location` is NOT valid (E1001).
- Valid optimization goals (SALES + WEB): `PIXEL_PURCHASE`, `PIXEL_SIGNUP`, `PIXEL_ADD_TO_CART`, `PIXEL_PAGE_VIEW`, `LANDING_PAGE_VIEW`. Do not add goals from other objectives ‚Äî they return E2844 with SALES objective.
- Ad squad pixel tracking: only `pixel_id` sent, always optional. `pixel_conversion_event` is NOT valid (E1001).
- Creative destination URL: `web_view_properties.url` (WEB_VIEW) or `deep_link_properties.deep_link_uri` (DEEP_LINK). `APP_INSTALL` creative type is not used and has been removed from `CreativeType` ‚Äî do not add it back (it was dead code with incorrect required fields).
- Ad destination URL: URL fields are NOT sent on the Ad payload ‚Äî Creative only. Ad payload: `ad_squad_id`, `creative_id`, `name`, `type`, `status`.
- Ad `type` for WEB_VIEW creatives is `"REMOTE_WEBPAGE"`. `AD_TYPE_MAP`: `WEB_VIEW ‚Üí REMOTE_WEBPAGE`, all others ‚Üí `SNAP_AD`.
- Interaction type is hardcoded to WEB_VIEW. **`call_to_action` is valid on `WEB_VIEW` creatives.** Do NOT send `call_to_action` on `SNAP_AD` creatives (E2002).
- `headline` is optional on creatives (`SnapCreativePayload`). Do NOT send `headline: ""` ‚Äî Snapchat defaults `brand_name` to `""` when omitted, and E2607 fires if both are the same value (including both empty). Orchestrator uses `cr.headline || undefined`.
- Batch error responses: errors in `sub_request_error_reason` (not `error_type`/`message`).
- Ad squad geo targeting: `targeting.geos` (NOT `geo_locations`) ‚Äî `{ country_code: string }` with **lowercase** codes. Old presets with `geoCountryCode` (singular) are migrated on load.
- Ad squad device targeting: `devices[].device_type` is `"MOBILE"` or `"WEB"`. Optional `os_type` (`"iOS"` or `"ANDROID"`) when MOBILE.
- Ad squad demographic targeting: `min_age` and `max_age` are **strings** (e.g. `"18"`, `"35+"`), not numbers. Sending numbers causes E1001.
- **`placement_v2` ‚Äî do not set after squad creation:** Snapchat rejects `placement_v2` in the POST body (causes squad creation to fail entirely). The orchestrator creates squads without `placement_v2` and leaves them in Snapchat's default `"CONTENT"` placement. Do NOT add a follow-up PATCH to set `placement_v2` ‚Äî once any `placement_v2` value (including `"CUSTOM"` or `"AUTOMATIC"`) is set on a squad, Snapchat returns E2025 ("Update is not supported for this entity") on all future PUTs, permanently locking the squad against budget/bid/status changes via the API. `placement_v2` is excluded from `ADSQUAD_PUT_ALLOWED_FIELDS` so it is never echoed back in PUT updates.

- Fields intentionally omitted from payloads: `frequency_cap_max_impressions`, `frequency_cap_time_period`, `shareable`. Hardcoded: `pacing_type` (`"STANDARD"`). `profile_properties: { profile_id: string }` is required on creatives (E2652 if absent, E2006 if null) ‚Äî orchestrator auto-fetches via `GET /api/snapchat/profiles`; returns early with errors if unresolvable.
- Batch API response order is not guaranteed ‚Äî orchestrator matches by `name` with positional-index fallback (`find(r => r.name === x) ?? results[i]`). Both layers required.
- **PUT `/adsquads/{id}` silently no-ops on read-only fields:** Snapchat returns HTTP 200 with `sub_request_status: "ERROR"` (and the unchanged adsquad echoed back) when the body contains server-computed fields like `created_at`, `updated_at`, `delivery_status`, `effective_status`, `forced_view_eligibility`, `auto_bid`, `ranking_score`. `updateAdSquad` strips to a whitelist (`ADSQUAD_PUT_ALLOWED_FIELDS` = id, campaign_id, name, type, status, targeting, delivery_constraint, billing_event, optimization_goal, bid_strategy, bid_micro, daily_budget_micro, lifetime_budget_micro, conversion_window, pacing_type, start_time, end_time, pixel_id) before PUT, and inspects `sub_request_status` on the response ‚Äî throws with `error_type: message` (or `sub_request_error_reason`) when not SUCCESS. Without both checks, the PATCH route returns 200 to the client while Snapchat never applied anything. **`placement_v2` is intentionally excluded** ‚Äî sending it back causes E2025 ("Update is not supported for this entity") on squads created with placement_v2. **`bid_micro: null` or `bid_micro: 0`** is excluded ‚Äî auto-bid squads return 0 from the API and sending it back triggers E2771 ("Bid is required on ad squad").
- **Stats API (`/adsquads/{id}/stats`):** `granularity=DAY` returns `timeseries_stats[0].timeseries_stat.timeseries[]` ‚Äî NOT `total_stats` (which is only used for `granularity=TOTAL`). The `spend` field in `timeseries_stats` is already in **micro-dollars** (do NOT multiply by 1,000,000). Times must be at midnight in the ad account's actual IANA timezone (passed from `SnapAdAccount.timezone` ‚Äî e.g. `"Asia/Jerusalem"`, `"America/Los_Angeles"`). Offset is computed dynamically via `Intl.DateTimeFormat` with `timeZoneName: "shortOffset"` ‚Äî handles half-hour offsets and DST automatically. `end_time` must be midnight of the day AFTER the last desired date (exclusive). `ts.start_time` in the response is UTC ‚Äî convert to local date via `Intl.DateTimeFormat("en-CA", { timeZone })` (not `slice(0,10)`) to avoid off-by-one for UTC+ zones. Sync route accepts a `force: boolean` param (default `false`) to bypass the 1-hour re-fetch throttle; date picker changes and manual refresh always pass `force: true`; historical window syncs pass `force: false`. Country breakdown (`report_dimension=country`) is not used ‚Äî stats are totals only. Valid `fields`: `impressions`, `swipes`, `spend`, `video_views`, `conversion_purchases`, `conversion_purchases_value`. **The value field is `conversion_purchases_value` (plural "purchases") ‚Äî NOT `conversion_purchase_value` (singular).** Using the singular name causes E1004 "Unknown Field" for all pixel-purchase squads. `conversion_purchases_value` is in micro-dollars (same unit as `spend`); stored in DB as `conversion_purchase_value` BIGINT (internal name) and divided by 1,000,000 in the combined/drilldown response mappers as `snap_purchase_value_usd`. `getAdSquadStats` uses a 3-tier fallback: (1) all six fields; (2) on E1004 for `conversion_purchases_value`, retry with base + `conversion_purchases`; (3) on another E1004, retry with base fields only ‚Äî so spend/impressions are never lost even for non-conversion squads. `conversion_purchases` is a count stored as `snap_results`. `snap_cost_per_result` is computed client-side as `spend_usd / snap_results`.
